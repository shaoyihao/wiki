    原理

    Git 本质上是一个可按内容寻址（content-addressable）的文件系统，基于此提供了版本控制的功能。

    Linus 所写的第一版 Git 其实就是实现了几个核心的底层命令，例如 cat-file、write-tree、read-tree 等。日常使用的 Git 命令通常是 commit、add、checkout、branch、remote 等上层命令，这些上层命令通过组合底层命令或直接操作底层数据对象，使 Git 底层实现细节对用户透明。

    一个项目文件夹就是一个工作区（Working Directory）。终端进入该文件夹，执行 git init 即可用 Git 管理该工作区，此时文件夹中多了一个隐藏文件夹 .git，称为版本库（Repository），用于存储所有用来维护、管理该项目的信息。

    版本库(.git)在工作区根目录下，那么 Git 命令一定要在工作区根目录下执行吗?

    当在 Git 工作区的某个子目录下执行命令时，会依次向上层查找 .git 目录。当找到 .git 时，就将其作为所操作的版本库，将其所在的目录作为工作区的根目录。

    在非 Git 工作区执行 Git 命令时会因为找不到 .git 目录而报错。

    其中 objects 文件夹是 Git 的功能基础，称为 Git 对象库。放在 objects 里的对象共有 4 种类型：blob（数据）、tree（目录）、commit（提交）、tag（标签），对项目的管理最终即落到对这些对象的管理。

    按内容寻址¶

    举个例子，通过 Git 底层命令 git hash-object 直接向对象库中存入数据（blob对象）：

    1
2
    echo "version 1" | git hash-object -w --stdin
# 83baae61804e65cc73a7201a7252750c76066a30

    写入后会返回一个长度为 40 位的 hash 值（SHA1），也即所存数据对应的“键”（也称为“对象ID”）。此时 objects 文件夹内容为：

    多出了一个文件，该文件存储在以新存入数据对应 hash 值的前 2 位命名的文件夹内，文件名为 hash 值的后 38 位（把太多文件放在同一个目录中，可能会导致文件系统变慢，将 SHA1 的首字节作为目录正是为了提高文件系统效率）。

    可见：Git 对象库逻辑上就是一个散列表，一个对象对应一条表项。

    SHA1

    SHA1是一种安全散列加密算法，发生“不同内容对应相同的 SHA1 值”的概率极小，因此可以认为这是一种一一映射。

    我们可以通过 git cat-file 这个 Git 底层命令查看某一键值（可以不必完全写出）对应的数据内容（不能用 cat 命令来查看，因为数据经过了二进制压缩）。

    1
2
3
4
    git cat-file -t  83baa    # -t选项用于查看键值对应数据的类型
# blob
git cat-file -p 83baa     # -p选项用于查看键值对应数据的内容
# version 1

    blob 对象¶

    在用 Git 管理项目文件时，一个 blob 对象就是一个文件的内容。将文件加入 Git 仓库，也就是为文件内容创建了副本。

    将文件 file.txt 中的内容（"version 1"）写入 Git 数据库：

    1
2
    git hash-object -w file.txt
# 83baae61804e65cc73a7201a7252750c76066a30  返回与之前一样的hash值

    由于 file.txt 中的内容之前已存入 Git 数据库，因此不会再存入，Git 数据库无变化。可见 Git 在存储对象时只关心对象的内容，而不关心文件名。如果两个文件的内容完全一样，无论是否在相同的目录，Git 在对象库里只保存一份 blob 类型的内容副本。

    对 file.txt 中的数据进行修改（"version 2"），再存入 Git 数据库：

    1
2
    git hash-object -w file.txt
# 1f7a7a472abf3dd9643fd615f6da379c4acb3e3a

    文件发生变化时，Git 会为它计算一个新的 SHA1 值，识别出它现在是一个不同的 blob 对象，然后把这个新的 blob 加到对象库里，原来的 blob 在对象库里保持不变。

    如果我们要将 file 的内容回退为 “version 1”，应该如何操作？可以这样：

    1
    git cat-file -p 83baa > file.txt

    虽然简单，但这其实大致反映了 Git 版本回退的底层原理。因为 Git 使用一个文件的全部内容的散列值作为键，因此它总是对完整文件进行操作，而不能将工作或对象库条目建立在文件内容的一部分或者文件的两个版本之间的差异上。

    tree 对象¶

    由目录文件可以构造一个 tree 对象。暂存区也可以视作一个文件夹，根据暂存区的内容可创建一个 tree 对象，存入 objects：

    1
2
    git write-tree
# 391a4e90ba882dbc9ea93855103f6b1fa6791cf6

    1
2
3
4
    git cat-file -t 391a4e
# tree
git cat-file -p 391a4e
# 100644 blob 83baae61804e65cc73a7201a7252750c76066a30  file.txt

    可见 tree 对象中包含文件名、文件类型、对象在 objects 中对应的索引等信息。

    暂存区发生变化时 index 被更新，且暂存区中包含的对象将被存入 objects（如果之前没有存过的话）。

    查看暂存区内容：

    git ls-files --stage

    commit 对象¶

    commit 对象用于记录什么时间由什么人因为什么原因提交了一个新的版本，及这个新的版本的父版本是什么。

    要创建一个 commit 对象，需要指定所需提交的 tree 对象的 ID，以及父 commit 对象（首次提交无需指定父对象）：

    1
2
    git commit-tree 19672 -m "first commit"
# ce9502e39d54d78515ec7ab383df716055bb7a8f

    commit 对象中包含的内容有：

    • 指向存储本次变化的 tree 对象的键
    • 作者、提交者的信息（名字、邮箱、时间、时区）
    • 提交说明

    版本追踪¶

    Git 版本库中每次存储的只是当前版本与前一个版本的不同之处，项目的若干次提交之间构成了一个有向无环图。

    引用¶

    在上述分析中，我们一直采用 hash 值来索引 objects 中存储的对象，这无疑是非常麻烦的，Git 提供了引用（reference）来简便操作。

    引用存储于 .git/refs 中，类似于指针，指向某一个对象，引用本身的值即所指对象的 hash 值。

    1
2
    echo "2ce2dbdb1866a176cb49d650baf611cc115d22fa" > .git/refs/heads/master
# 创建 master 指向最新的 commit 对象

    之后便可以用 “master” 替换 “2ce2dbdb1866a176cb49d650baf611cc115d22fa”。

    Git 内置了一些特殊引用：

    • main 通常指向主分支的最新一次提交；
    • HEAD 指向当前分支的最近一次提交；
    • HEAD^ 指向前一次提交（父结点）；
    • HEAD~n 指向前 \(n\) 次提交
